# 캐시 계획

Toycore의 초기 코어는 캐시 없이 동작해야 합니다. 캐시는 필수 의존성이 아니라 성능 최적화 계층입니다.

저가형 웹호스팅을 지원해야 하므로 Redis, Memcached, 별도 캐시 서버, 상시 실행 워커를 전제로 하지 않습니다. 캐시를 도입하더라도 파일 기반 캐시부터 시작합니다.

## 결론

```text
초기 코어: 캐시 없이 동작
1차 캐시 대상: 사이트 설정, 활성 모듈 목록, 번역 배열
HTML 전체 캐시: 코어 기본 기능이 아니라 공개 콘텐츠 모듈의 선택 기능
외부 캐시 서버: 기본 의존성으로 두지 않음
```

## 캐시가 없어도 동작해야 하는 이유

- 설치와 디버깅이 단순해야 함
- 저가형 웹호스팅에서 파일 권한 문제가 생길 수 있음
- 캐시 무효화 정책이 늦게 정해지면 데이터 불일치가 생김
- 로그인, CSRF, locale, 권한, 개인정보 화면은 잘못 캐시하면 보안 문제가 됨

따라서 1단계 구현은 DB와 파일만으로 명확히 동작해야 합니다.

## 코어 캐시 대상

초기 이후 코어가 캐시할 수 있는 대상은 제한합니다.

### 사이트 설정

대상:

- `toy_sites`
- `toy_site_settings`
- 사이트 기본 locale
- 사이트 기본 SEO 설정
- 홈 제공자 설정

예시 파일:

```text
storage/cache/site_1_settings.php
```

예시 내용:

```php
<?php

return [
    'site_id' => 1,
    'name' => 'Toycore',
    'default_locale' => 'ko',
    'home.module' => 'page',
    'home.action' => 'home',
];
```

### 활성 모듈 목록

대상:

- `toy_modules`
- `toy_site_modules`

예시 파일:

```text
storage/cache/site_1_modules.php
```

### 번역 배열

대상:

- `lang/{locale}/core.php`
- `modules/{module_key}/lang/{locale}.php`

초기에는 PHP 파일을 직접 include해도 충분합니다. 번역 파일 수가 늘어나면 병합 캐시를 검토합니다.

## 코어 캐시에서 제외하는 것

다음은 코어 기본 캐시 대상에서 제외합니다.

- 로그인 상태에 따라 달라지는 HTML
- 관리자 화면
- 회원 전용 화면
- CSRF 토큰이 포함된 폼
- 개인정보가 포함된 화면
- 장바구니, 주문, 계정 화면
- 권한에 따라 출력이 달라지는 화면
- 사용자별 HTML 조각

## HTML 전체 캐시

HTML 전체 캐시는 초기 코어 기능으로 넣지 않습니다.

HTML 캐시가 필요한 경우 다음 조건을 만족하는 공개 콘텐츠 모듈이 자체적으로 구현합니다.

- 비로그인 사용자에게 동일하게 보이는 페이지
- CSRF 토큰이 없음
- 개인정보가 없음
- 권한별 출력 차이가 없음
- locale별 캐시 키가 분리됨
- 콘텐츠 수정 시 캐시를 무효화할 수 있음

예:

```text
page 모듈: 공개 페이지 HTML 캐시 선택 구현
board 모듈: 공개 게시글 상세 HTML 캐시 선택 구현
shop 모듈: 상품 상세 HTML 캐시 선택 구현
```

## 캐시 키 기준

캐시를 만들 때는 최소한 다음 값을 고려합니다.

```text
site_id
locale
module_key
path 또는 content_id
updated_at 또는 version
```

회원 ID, 권한, 동의 상태가 필요한 화면은 HTML 전체 캐시 대상에서 제외합니다.

## 무효화 정책

캐시는 쓰기 작업 이후 명시적으로 삭제하는 방식을 우선합니다.

예:

```text
사이트 설정 변경 -> site settings cache 삭제
모듈 활성화 변경 -> module cache 삭제
번역 파일 변경 -> lang cache 삭제
페이지 수정 -> page 모듈의 해당 페이지 캐시 삭제
```

저가형 웹호스팅을 고려해 파일 잠금 실패나 캐시 디렉터리 쓰기 실패가 있어도 사이트가 죽지 않아야 합니다. 캐시 쓰기에 실패하면 DB 또는 원본 파일을 직접 읽습니다.

## 저장 위치

캐시 파일은 웹 루트 밖에 둡니다.

권장 위치:

```text
storage/cache/
```

`storage/cache`가 웹에서 직접 접근 가능한 위치에 있다면 `.htaccess` 또는 서버 설정으로 접근을 차단해야 합니다.

## 구현 단계

1. 캐시 없이 코어 구현
2. 설정 조회 helper에 파일 캐시 선택 적용
3. 활성 모듈 목록 캐시 적용
4. 번역 배열 병합 캐시 검토
5. 공개 콘텐츠 모듈에서 HTML 캐시 선택 구현

## 금지하는 방향

초기 구현에서는 다음을 도입하지 않습니다.

- Redis 필수 의존
- Memcached 필수 의존
- 프레임워크형 캐시 레이어
- 모든 DB 쿼리 자동 캐시
- 사용자별 전체 HTML 캐시
- 관리자 화면 HTML 캐시
- CSRF 폼 HTML 캐시
