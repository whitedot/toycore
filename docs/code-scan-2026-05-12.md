# 코드 정밀 스캔 결과

기준일: 2026-05-12

## 목적

Toycore 전체 코드를 보안, 요청 흐름, 모듈 경계, DB 접근, 설치/업데이트 흐름 관점에서 단계별로 확인하고, 즉시 수정할 항목과 후속 개선 후보를 분리한다.

## 스캔 단계

### 1. 기준 문서 확인

확인 문서:

- `docs/security-checklist.md`
- `docs/database-access-policy.md`
- `docs/module-checklist.md`
- `docs/module-update-policy.md`
- `docs/core-decisions.md`

판정 기준:

- 요청 흐름은 명시적 include와 request contract를 따라야 한다.
- 상태 변경 POST는 CSRF 검증을 통과해야 한다.
- 관리자 action은 로그인과 role 검증을 통과해야 한다.
- 동적 SQL 값은 prepared statement를 사용해야 한다.
- 식별자 조립이 필요한 경우 allowlist 또는 안전한 table name 검증을 거쳐야 한다.
- 모듈 설치/활성화/업데이트는 메타데이터, 계약 파일, 의존성, route 충돌, 버전 차이를 구분해야 한다.

### 2. 자동 점검 실행

실행 명령:

```sh
./.tools/bin/check
```

결과:

- 통과.
- `git diff --check` 통과.
- retention/auth/runtime/storage/output/member/admin/community/popup 관련 정책 스크립트 통과.
- 전체 PHP lint 통과.

비고:

- 시스템 `php` 명령은 없지만 `.tools/bin/php`가 Docker 기반 PHP 8.3.30 런타임을 제공한다.

### 3. 요청 흐름과 redirect 스캔

실행한 주요 패턴:

```sh
rg -n "header\s*\(\s*['\"]Location:" core modules index.php
rg -n "\b(exit|die)\s*(\(|;)" modules core index.php
rg -n "\$_SERVER\['REQUEST_URI'\]|\$_REQUEST|\$_COOKIE|\$_FILES" core modules index.php
rg -n "include \$|require \$|include\s+\$|require\s+\$" core modules index.php
```

결과:

- `header('Location')` 직접 호출은 코어 redirect helper에만 남아야 한다.
- `index.php`와 `toy_finish_response()` 계층의 `exit` 사용은 요청 진입/응답 종료 경계로 허용한다.
- 동적 include 의심 패턴은 발견되지 않았다. 커뮤니티 skin/theme include는 helper가 allowlist로 결정한 파일을 include하는 구조로 별도 검토 대상이다.
- `REQUEST_URI`, `REQUEST`, `COOKIE` 직접 의존 패턴은 발견되지 않았다.

수정:

- `modules/banner/helpers.php`의 외부 배너 링크 redirect가 직접 `header('Location')`를 호출하던 것을 `toy_redirect_external()` 호출로 변경했다.

### 4. DB 접근과 식별자 조립 스캔

실행한 주요 패턴:

```sh
rg -n "ORDER BY .*\$|FROM ' \\.|JOIN ' \\.|UPDATE ' \\.|INSERT INTO ' \\.|DELETE FROM ' \\." core modules
rg -n "->query\(|->exec\(" core modules
```

결과:

- 대부분의 `query()`는 고정 SQL 또는 존재 확인용 고정 쿼리다.
- `core/helpers/ledger.php`는 table name을 조립하지만 `toy_` table allowlist를 통과한 뒤 사용한다.
- `modules/admin/helpers/dashboard.php`는 table 존재 확인에서 `toy_` table allowlist를 통과한 뒤 식별자를 조립한다.
- `core/helpers/sql.php`의 `exec()`는 설치/업데이트 SQL 파일 실행 전용이다.

후속 관찰:

- `ledger`는 `docs/core-decisions.md`의 공유 도메인 패턴 helper 감시 대상이다. 현재는 좁은 DB primitive에 머물지만, 정책이나 조회 UI가 붙으면 모듈 또는 별도 선택 모듈로 분리해야 한다.

### 5. 모듈 lifecycle 스캔

확인 범위:

- `/admin/modules`
- `/admin/updates`
- `.tools/bin/check.php`
- `docs/module-update-policy.md`
- `docs/smoke-test.md`

결과:

- 설치/활성화/업데이트 완료 판정 기준이 문서화되어 있다.
- `/admin/modules`는 설치 미완료, 계약 오류, SQL 적용 필요, 파일 전용 업데이트 가능, 코드 버전 낮음, 활성 최신, 비활성 최신 상태를 구분한다.
- 활성화 전 코드 버전이 설치 버전보다 낮은 경우 차단한다.
- 파일 전용 버전 반영은 pending SQL이 없고 계약/의존성 검증을 통과한 모듈만 대상으로 한다.
- `.tools/bin/check.php`가 lifecycle UI와 메타데이터 안전장치 존재를 확인한다.

## 즉시 수정한 항목

### 배너 외부 링크 redirect helper 정리

파일:

- `modules/banner/helpers.php`

내용:

- 직접 `toy_enforce_request_contract()`, `header('Location')`, `toy_finish_response()`를 호출하던 외부 링크 redirect를 `toy_redirect_external()`로 변경했다.

효과:

- redirect 응답 종료 경로가 코어 helper로 모인다.
- 요청 contract와 URL 검증 정책이 코어 redirect helper와 일관된다.

## 후속 개선 후보

### 1. 커뮤니티 skin/theme include 경계 주석 또는 검사 보강

현재 커뮤니티 public 화면은 `$skinView`, `$themeView`를 include한다. helper에서 allowlist와 파일 존재 검증을 거치는 구조라 즉시 결함으로 보지는 않는다.

다만 동적 include 패턴으로 보이기 쉬우므로, 정적 검사에 `theme/skin helper를 통한 include만 허용` 같은 규칙을 추가하면 좋다.

### 2. dashboard count SQL helper의 입력 범위 문서화

`toy_admin_dashboard_count($pdo, $sql)`은 현재 내부 고정 SQL만 넘긴다. 외부 입력은 섞이지 않는다.

후속으로 helper 이름이나 주석에서 "고정 SQL 전용"을 명시하면 DB 접근 정책과 더 잘 맞는다.

### 3. ledger helper 분리 여부 주기 검토

`core/helpers/ledger.php`는 포인트/예치금/적립금 반복을 줄이는 공유 도메인 패턴이다.

현재 유지 조건은 만족하지만, 다음 변화가 생기면 분리한다.

- 거래 취소, 만료, 정산, 지급 정책이 코어 helper에 추가됨
- 한 모듈 요구에 맞춘 컬럼이나 로직이 들어감
- 조회 UI 또는 관리자 workflow가 코어로 들어옴

## 현재 결론

이번 스캔 기준으로 즉시 수정이 필요한 항목은 배너 외부 링크 redirect helper 정리 1건이었다.

자동 점검과 PHP lint는 통과했으며, 나머지는 후속 개선 후보로 관리한다.
