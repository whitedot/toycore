# 레거시 운영자 관점 개선 계획

이 문서는 오래 운영한 PHP 프로젝트를 다뤄 온 개발자가 Toycore를 인수하거나 검토할 때 우선적으로 개선하고 싶어할 지점을 정리한다. 목표는 프레임워크식 전환이 아니라, 현재의 절차형 구조를 유지하면서 운영 중 사고와 수정 부담을 줄이는 것이다.

## 판단 기준

- 요청 흐름은 파일을 열어 확인할 수 있어야 한다.
- 저가형 공유호스팅에서도 설치, 백업, 복구 경로가 설명되어야 한다.
- 큰 action 파일은 URL과 권한 흐름이 흐려지기 전에 의도별 처리 단위로 분리한다.
- 공통 폼 처리와 감사 로그는 추상화보다 일관성을 우선한다.
- 업데이트와 모듈 파일 교체는 실패 후 재시도 기준이 먼저 보여야 한다.

## 개선 순서

### 1단계: 비대해지는 action 파일 정리

관리자 action은 권한 확인, 요청 처리 위임, 화면 데이터 준비, view include 흐름만 남긴다. 긴 POST intent 처리와 목록 데이터 조립은 이름이 분명한 helper로 옮기되, 라우팅 구조와 폼 필드는 바꾸지 않는다.

완료 기준:

- `modules/admin/actions/modules.php`에서 전체 요청 흐름을 한 화면에서 파악할 수 있다.
- 모듈 설치, 소스 반영, 상태 변경, 설정 변경의 기존 동작이 유지된다.
- PHP 문법 검사와 기본 점검 스크립트가 통과한다.

### 2단계: 관리자 POST 처리 규칙 정리

관리자 action마다 반복되는 `intent`, CSRF, `$errors`, `$notice`, 권한 확인, 감사 로그 기록 규칙을 문서화한다. helper를 만들더라도 요청의 실제 분기는 action 또는 action 전용 helper에서 명시적으로 보이게 둔다.

완료 기준:

- 새 관리자 POST action 작성 시 따라야 할 최소 순서가 문서에 있다.
- 오류 메시지와 성공 안내의 변수 이름, view 전달 방식이 일관된다.
- 감사 로그에 남길 값과 남기면 안 되는 값이 분리되어 있다.

### 3단계: 설치/업데이트 실패 복구 UX 강화

설치, 모듈 파일 교체, SQL 업데이트는 운영 장애 가능성이 큰 흐름이다. 실패 marker, 파일 로그, 감사 로그, 백업 디렉터리, 재시도 기준을 관리자와 문서에서 같은 용어로 설명한다.

완료 기준:

- 실패 후 운영자가 먼저 볼 파일과 화면이 문서화되어 있다.
- SQL 일부 적용, 파일 일부 교체, checksum 불일치의 처리 기준이 분리되어 있다.
- 백업 디렉터리 보관과 정리 정책이 있다.

### 4단계: 공유호스팅 배포 체크리스트 보강

공유호스팅에서는 `.htaccess`, 문서 루트, PHP extension, 파일 권한, 업로드 제한이 사이트마다 다르다. 설치 전 차단 확인과 대체 방법을 운영 문서에서 더 구체적으로 제공한다.

완료 기준:

- `config/`, `storage/`, `database/`, `modules/` 직접 접근 차단 확인 항목이 있다.
- `ZipArchive`가 없을 때의 운영 제한이 설명되어 있다.
- `storage/` 쓰기 권한과 로그 파일 생성 기준이 있다.

### 5단계: 스모크 테스트 확대

전체 테스트 프레임워크보다 설치와 운영의 핵심 흐름을 빠르게 확인하는 스모크 테스트를 우선한다.

완료 기준:

- PHP 문법 검사, SQL 파일 검증, 모듈 구조 검증은 기본으로 유지한다.
- 설치, 로그인, 관리자 모듈 화면, 업데이트 확인까지 최소 HTTP 검증 시나리오를 문서화한다.
- Docker가 없는 Windows 개발 환경에서도 실행 가능한 점검 경로를 유지한다.

## 이번 반영 범위

- 1단계의 첫 작업으로 관리자 모듈 관리 action을 얇게 정리한다.
- 3단계와 4단계는 운영 문서에 즉시 필요한 확인 항목을 추가한다.
- 2단계와 5단계는 다음 변경에서 실제 공통 규칙과 스모크 시나리오로 이어간다.
